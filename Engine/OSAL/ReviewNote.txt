HARDBOP ENGINE CODE REVIEW - OSAL Module
================================================================================

OVERVIEW: OS Abstraction Layer - platform-specific implementations.

FILES:
  - OSThread.h, OSInputOutput.h, OSMemory.h, OSDebug.h
  - Platform-specific: Windows*.h, OSX*.h, Linux*.h
  - Intrinsic.h - Compiler intrinsics (likely, unlikely, debugBreak)
  - Directory.h, File.h - File system abstractions
  - SourceLocation.h - Source location capture

STRENGTHS:
  + Clean platform abstraction
  + Intrinsics properly handle clang, GCC, MSVC
  + Platform-specific implementations in separate files
  + File and Directory abstractions

CRITICAL ISSUES:
  - CRITICAL: VirtualFree on Linux/macOS uses free() instead of munmap()
    Files: OSXMemory.cpp:29, LinuxMemory.cpp:27
    Impact: Memory corruption - mmap'd memory must be freed with munmap, not free
    Fix: Replace free(address) with munmap(address, 0)
    
  - CRITICAL: VirtualFree on Windows has inverted error check
    Files: OSMemory.cpp:112, WindowsMemory.cpp:37
    Issue: Checks if (result) instead of if (!result) - triggers Assert on SUCCESS
    Impact: Engine crashes on successful memory deallocation
    
  - CRITICAL: VirtualFree in OSMemory.cpp Linux branch references undefined 'size'
    File: OSMemory.cpp:28
    Issue: Parameter unnamed but body uses 'size' variable
    Impact: Should not compile - compilation error

MEDIUM ISSUES:
  - GetBackTrace() on Linux returns "Not Implemented" (LinuxDebug.cpp:9)
  - Dual memory implementation files may cause ODR violations
    Issue: Both OSMemory.cpp (uses __linux__ etc.) and platform-specific files exist
  - OSThreadTest::Prepare() is empty with TODO comment (OSThread.cpp:14-17)
