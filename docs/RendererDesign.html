HardBop Engine — RHI Architecture

Architecture: Hybrid Pipeline (Extract → Prepare → Render Graph → Submit)
Core Principle: Bindless, Stateless, and Data-Oriented.
Shader Philosophy: HLSL as Single Source of Truth (SM 6.6+).

1. Fundamentals of Modern Graphics APIs

Modern APIs (Vulkan, DX12, Metal) represent a paradigm shift from the "State Machine" model used in OpenGL/DX11 to an "Explicit Control" model. This shift is driven by the need to reduce driver overhead and utilize the massive parallelism of modern GPUs.

Key Design Pillars

  • Explicit Resource Management: The application, not the driver, is responsible for allocating, pooling, and managing the lifetime of buffers and textures. This eliminates "hidden" driver-side memory reallocations.
  • Synchronization & Barriers: In older APIs, the driver implicitly handled dependencies. In modern APIs, the programmer must explicitly define Pipeline Barriers and Memory Barriers to ensure that a write operation (e.g., a Compute Shader) is visible to a subsequent read operation (e.g., a Fragment Shader).
  • Bindless Rendering: Instead of binding resources to "slots" (e.g., `texture0`, `texture1`), resources reside in massive, globally accessible arrays (Descriptor Heaps/Sets/Argument Buffers). Shaders access them via 32-bit indices, drastically reducing CPU-side driver overhead and state changes.
  • Pipeline State Objects (PSOs): All shader permutations, blend states, and rasterizer states are pre-baked into immutable PSOs. This prevents the "shader stutter" famously seen in modern title launches when the driver tries to compile a state on-the-fly during a draw call.

2. Hybrid Pipeline Architecture

The HardBop Engine implements a 4-phase pipeline designed for high-throughput, data-oriented execution.

Pipeline Execution Flow

EXTRACT
PREPARE
GRAPH
SUBMIT

Data evolves from ECS World $\rightarrow$ SoA Buffers $\rightarrow$ Culled Indices $\rightarrow$ GPU Command Buckets

Phase 1: EXTRACT

Lockless extraction of ECS world data into CPU-side Structure-of-Arrays (SoA) buffers. This phase transforms fragmented entity data into contiguous, cache-friendly memory blocks.

Phase 2: PREPARE

The "Compute-Heavy" phase. Compute shaders execute frustum culling, Hi-Z occlusion rejection, and LOD selection. The output is a highly refined set of visible instance identifiers.

Phase 3: RENDER GRAPH

A Directed Acyclic Graph (DAG) of rendering passes. The engine automatically resolves dependencies and injects the necessary API-specific barriers (VK Pipeline Barriers, DX12 Enhanced Barriers, etc.).

Phase 4: SUBMIT

Command buckets are sorted using a 64-bit radix sort key to minimize pipeline state changes. The final, sorted command stream is submitted to the hardware with zero lock contention.

64-bit Sort Key Layout

The 64-bit key is designed for high-performance Radix Sorting:

Bits Component Description
63 PassID Primary grouping (Graphics vs Compute)
59-48 PSO Hash Minimizes Pipeline State changes
47-32 Material Index Group by Material/Texture bindless set
31-0 Depth/Distance Z-sorting for transparency/efficiency

6. Critical Edge Cases

Metal Tier 2 Limits

Metal hardware has stricter limits on the number of textures per argument buffer. The engine implements Descriptor Partitioning to split global indices into manageable pages.

Temporal History Aliasing

To prevent motion artifacts in TAA/DLSS, resources marked as HistoryCritical are protected from aliasing and premature reclamation during the Render Graph execution.
