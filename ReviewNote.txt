                            HARDBOP ENGINE CODE REVIEW

================================================================================
DIRECTORY: Engine/Config
================================================================================
OVERVIEW: Configuration system with key-value parameters and file-based settings.

FILES:
  - BuildConfig.h     - Build-time configuration flags
  - ConfigParam.h    - Thread-safe config parameter template
  - ConfigSystem.h   - Global config registry
  - ConfigFile.h     - File-based configuration
  - EngineConfig.h  - Engine-specific settings

OBSERVATIONS:
  + Good: Template-based ConfigParam with optional atomic support
  + Good: Conditional compilation for parameter descriptions (ENGINE_PARAM_DESC_ENABLED)
  + Good: Thread ID tracking in debug builds for thread safety validation
  ~ Note: std::mutex in ConfigParam may cause contention in high-frequency access
  - Issue: Constructor uses conditional initialization - could use delegating constructors

STYLE CONCERNS:
  - Mix of camelCase and PascalCase in internal types (TValue vs template params)

================================================================================
DIRECTORY: Engine/Container
================================================================================
OVERVIEW: Core container implementations with custom allocator support.

FILES:
  - Array.h          - Static array with allocator template
  - LinkedList.h     - Linked list implementation
  - AtomicStackView.h - Lock-free stack

OBSERVATIONS:
  + Excellent: Array uses placement new for proper object construction
  + Excellent: Move semantics properly implemented (move constructor, move assignment)
  + Good: FatalAssert for bounds checking in debug builds
  + Good: Custom iterator support with begin()/end()
  + Excellent: Proper RAII destructor with element destruction

PATTERNS:
  - Array uses swap-and-clear idiom for Clear()
  - Allocator type is template parameter with DefaultAllocator as default

  - Task.h               - Task definition
  - ComponentSystem.h    - Entity-component system

OBSERVATIONS:

  + Good: Branch prediction hints (likely/unlikely) in Debug.h
  + Good: Compile-time guards for debug-only code
  - Issue: Exception.h exists but project claims exception-free
  - Issue: CommandLineArguments - verify it's used correctly

KEY PATTERNS:
  - StaticString for type-safe names

  - Task affinity system for load balancing

  - Matrix2x2.h, Matrix3x3.h, Matrix4x4.h - Matrix classes

PATTERNS:

================================================================================
                                OVERALL SUMMARY
================================================================================

STRENGTHS:
  1. Comprehensive memory allocator system for different use cases
  2. Clean module organization (Config, Container, Core, Memory, Math, etc.)
  3. Good use of C++23 features (concepts, static_assert, std::atomic)
  4. Proper RAII and move semantics throughout
  5. Consistent naming conventions (PascalCase for types, hbe namespace)
  6. Platform abstraction with OSAL
  7. Async logging system
  8. Template-based flexible numeric types in Math
  9. String interning for performance

AREAS FOR IMPROVEMENT:
  1. Exception.h exists but project claims exception-free - verify consistency
  2. Some CMakeLists.txt files have duplicate entries
  3. ConfigParam constructor could use delegating constructors
  4. PreEngineInit pattern in Engine is unusual
  5. Consider adding more [[nodiscard]] to allocation functions
  6. Some files have inconsistent inline placement (some inline in class, some outside)

CODING STANDARDS COMPLIANCE:
  + Mostly follows .clang-format settings
  + Uses #pragma once for header guards
  + Proper use of [[nodiscard]], [[maybe_unused]]
  + Good separation of definition blocks
  + Consistent import ordering (system, then local)
